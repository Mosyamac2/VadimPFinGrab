# Техническое задание

## ETL-пайплайн извлечения финансовой отчётности с e-disclosure.ru с использованием LLM

**Версия:** 1.0
**Дата:** 2026-05-01
**Статус:** согласовано

---

## 1. Цель и контекст

Создать локально работающий пайплайн, который автоматически собирает финансовую отчётность Top-50 эмитентов Московской биржи с портала **e-disclosure.ru**, извлекает из PDF-отчётов набор ключевых финансовых показателей (с использованием LLM как извлекающего движка), валидирует их и публикует результат в виде Excel-файла на Google Drive. Параллельно собирается и структурируется поток сообщений о существенных фактах. Запуск выполняется по расписанию (раз в сутки) и по требованию (кнопка "обновить").

Архитектурно пайплайн должен быть подготовлен к последующему расширению до клиент-серверной схемы с доступом через мобильное приложение iOS — это требование к организации кода, а не функциональное требование первой версии.

---

## 2. Источник данных

- **Основной источник:** `https://www.e-disclosure.ru/` — Центр раскрытия корпоративной информации.
- **Тип контента:** RAR-архивы с пакетами раскрытия эмитента (содержат PDF-отчётность и сопроводительные документы), отдельные сообщения о существенных фактах.
- **Способ получения:** HTTP-скрейпинг карточек эмитентов и страниц публикаций (без публичного API на момент составления ТЗ — требуется проверить актуальность).
- **Авторизация:** не требуется (публичная информация).
- **Соблюдение нагрузки:** ограничение частоты запросов (rate limit), User-Agent, паузы между запросами; уважение `robots.txt`.

---

## 3. Перечень эмитентов

- Конфигурируется через файл `config/tickers.yaml` (формат — список тикеров MOEX и/или ОГРН/ИНН/наименований эмитентов с маппингом на e-disclosure ID).
- Дефолтное наполнение: **Top-50 эмитентов MOEX по капитализации** на момент инициализации проекта (формируется однократно вручную и фиксируется в YAML).
- Список **статичный** — обновление состава осуществляется правкой конфига. Динамическое подтягивание состава индекса в первой версии не реализуется.

---

## 4. Глубина истории и инкрементальность

- **Первоначальная загрузка (backfill):** последние **3 календарных года** отчётности на момент первого запуска.
- **Регулярный запуск:** **инкрементальный режим** — обрабатываются только публикации с датой публикации позже даты последней успешной обработки соответствующего эмитента (хранится в state-БД).
- **Кнопка "обновить"** инициирует тот же инкрементальный цикл, но вне расписания.
- **Полная переобработка** доступна как отдельная команда CLI (`--full-reload`) — на случай ошибок в логике извлечения и необходимости пересобрать витрину.

---

## 5. Извлекаемые финансовые показатели

### 5.1. Приоритезация типов отчётности

При наличии у эмитента нескольких типов отчётности за один и тот же отчётный период применяется приоритет:

1. МСФО (IFRS) — приоритет 1
2. РСБУ — приоритет 2 (берётся, только если МСФО отсутствует)

Приоритет фиксируется в конфиге и может быть изменён.

### 5.2. Список показателей (по умолчанию)

- Выручка (Revenue)
- EBITDA
- Чистая прибыль (Net Income)
- Активы (Total Assets)
- Долг (Total Debt — финансовый долг, кратко- и долгосрочный)

### 5.3. Расширяемость

Список показателей задаётся в `config/metrics.yaml`. Каждый показатель описывается:

- Каноническое имя (используется в Excel-витрине).
- Синонимы / варианты наименования в РСБУ и МСФО (для повышения точности извлечения LLM).
- Единица измерения (валюта, тыс./млн).
- Опциональная формула вычисления из других показателей (например, EBITDA через Net Income + D&A + Interest + Tax) — на случай, если показатель отсутствует в отчёте напрямую.

Добавление нового показателя выполняется правкой YAML без изменения кода пайплайна.

---

## 6. Сообщения о существенных фактах

- Собираются **все типы** существенных фактов, публикуемых эмитентом на e-disclosure.
- Каждое сообщение прогоняется через LLM с задачей структурированного извлечения:
  - тип события (классификация по справочнику: дивиденды, смена менеджмента, M&A, существенная сделка, корпоративное действие, изменение в составе акционеров, прочее);
  - дата события;
  - дата публикации;
  - краткое резюме (1–3 предложения);
  - ключевые числовые параметры (если применимо: сумма сделки, размер дивиденда и т.п.);
  - ссылка на оригинал на e-disclosure.
- Результат пишется в **отдельный лист** Excel-витрины: `events`.
- Справочник типов событий — конфигурируемый (`config/event_types.yaml`).

## 7. Архитектура

Пайплайн строится как последовательность изолированных модулей с явными контрактами вход/выход. Каждый модуль вызывается через общий оркестратор и логирует свой статус в state-БД.

### 7.1. Стадии пайплайна

1. **Discoverer** — обходит карточки эмитентов из конфига, формирует список новых публикаций (отчёты + существенные факты) с момента последней обработки.
2. **Downloader** — скачивает архивы и отдельные документы, складывает в локальное blob-хранилище (`data/raw/{ticker}/{publication_id}/`).
3. **Unpacker** — распаковывает RAR/ZIP, инвентаризирует содержимое, выделяет PDF-кандидаты на парсинг.
4. **PDF Classifier** — для каждого PDF определяет:
   - является ли он формой отчётности нужного типа (МСФО / РСБУ / прочее);
   - является ли он машиночитаемым (есть текстовый слой) или сканом (требуется OCR);
   - тип отчётной формы (баланс, ОФР, пояснения и т.д.).
5. **Text Extractor** — извлекает текст из PDF:
   - для машиночитаемых — нативное извлечение (`pdfplumber` / `pymupdf`);
   - для сканов — OCR (Tesseract `rus+eng` по умолчанию; альтернатива через конфиг — облачный OCR).
6. **Metric Extractor (LLM)** — на вход подаётся очищенный текст (или, где поддерживается, нативный PDF), на выходе — структурированный JSON с показателями из `config/metrics.yaml` за все отчётные периоды, обнаруженные в документе. Использует строгую JSON-схему ответа.
7. **Event Extractor (LLM)** — аналогично для сообщений о существенных фактах, со своей JSON-схемой.
8. **Validator** — sanity checks (см. раздел 11).
9. **Writer** — записывает результат в SQLite-витрину и реплицирует Excel на Google Drive.

### 7.2. Принципы разделения

- Каждая стадия — отдельный Python-модуль с функцией `run(input) -> output`, не зависящей от состояния других стадий.
- Оркестратор хранит DAG стадий и состояние выполнения в SQLite.
- Любая стадия может быть запущена изолированно (для отладки и переобработки).
- Внешние интеграции (LLM-провайдер, OCR-движок, Google Drive, источник e-disclosure) подключаются через интерфейсы-абстракции — это даёт возможность подменить провайдера без правки бизнес-логики.

### 7.3. Подготовка к мобильному доступу (на будущее)

- Бизнес-логика отделена от способа запуска: ядро пайплайна не знает, вызвано ли оно из CLI, по cron'у или из HTTP-эндпоинта.
- Состояние и витрина хранятся в SQLite + Excel на Google Drive — оба формата читаемы из мобильных клиентов сторонними средствами (Numbers, Excel iOS, любые SQLite-вьюверы).
- В будущем над ядром можно поднять FastAPI-обёртку и на её базе — мобильное приложение. Это **вне scope первой версии**, но архитектура должна это допускать без рефакторинга ядра.

## 8. Стек технологий

| Компонент | Выбор | Обоснование |
|---|---|---|
| Язык | Python 3.11+ | Зрелые библиотеки для PDF, OCR, скрейпинга, LLM SDK |
| HTTP-клиент | `httpx` (async) | Производительный async-скрейпинг с лимитом по rate |
| Парсер HTML | `selectolax` или `BeautifulSoup` | По вкусу разработчика |
| Распаковщик RAR | `rarfile` (требует системный `unrar`) | RAR не open-source, без unrar никак |
| Парсер PDF (нативный) | `pdfplumber` + `pymupdf` (fitz) | Первый — для таблиц, второй — для текста и быстроты |
| OCR (локальный) | Tesseract 5 с `rus+eng` traineddata | Бесплатно, оффлайн, приемлемое качество для нативно отсканированных отчётов |
| OCR (опциональный, облачный) | Yandex Vision OCR / Google Vision | Подключается через конфиг, выше качество, но платно и с передачей данных наружу |
| LLM-провайдер | Anthropic Claude (Sonnet 4.6) | Лучшее качество структурированного извлечения из финансовых документов на момент составления ТЗ |
| LLM-доступ | Двухканальный: прямой Anthropic API (приоритет, если ключ задан) + OpenRouter (fallback) | Прямой API лучше поддерживает нативный PDF-input, OpenRouter — для гибкости |
| Excel-генерация | `openpyxl` | Полный контроль над форматированием |
| State-БД | SQLite | Один файл, без сервера, читается отовсюду |
| Облачное хранилище | Google Drive API v3 | По требованию заказчика |
| Аутентификация Google | OAuth 2.0 (refresh token) | Локальный запуск с личного аккаунта |
| Планировщик | системный `cron` (Linux) с CLI-вызовом пайплайна; в качестве альтернативы — `systemd` timer | Целевая ОС — Linux; нет необходимости держать long-running процесс |
| Логирование | `structlog` | Структурированные логи в JSON для последующего анализа |
| Конфигурация | YAML + Pydantic-валидация | YAML удобен для редактирования человеком, Pydantic ловит ошибки на старте |

## 9. Конфигурация

Все настройки вынесены в файлы в каталоге `config/` и валидируются Pydantic-схемами при старте.

### 9.1. Список файлов

- `config/app.yaml` — общие настройки (пути, расписание, режимы).
- `config/tickers.yaml` — список эмитентов с маппингом на e-disclosure ID.
- `config/metrics.yaml` — извлекаемые показатели и их синонимы.
- `config/event_types.yaml` — справочник типов существенных фактов.
- `config/llm.yaml` — провайдеры LLM, модели, лимиты.
- `config/ocr.yaml` — настройки OCR-движка.
- `.env` (вне Git) — секреты: API-ключи, OAuth-токены.

### 9.2. Принципы

- Все строковые литералы, влияющие на бизнес-логику (имена показателей, типы событий, приоритеты типов отчётности), вынесены в конфиг.
- Никаких "магических чисел" в коде.
- Изменение конфига не требует пересборки или перезапуска при использовании кнопки "обновить" — конфиг перечитывается на старте каждого цикла.

## 10. Хранилище данных и состояния

### 10.1. Локальная структура

```
project_root/
├── config/                  # YAML-конфиги (см. раздел 9)
├── data/
│   ├── raw/                 # скачанные архивы и распакованные PDF
│   ├── processed/           # промежуточные текстовые extracts
│   └── state.sqlite         # состояние пайплайна и витрина
├── output/
│   └── e-disclosure.xlsx    # локальная копия Excel-витрины
├── logs/
└── src/
```

### 10.2. Схема state-БД (SQLite)

- `tickers` — справочник эмитентов.
- `publications` — все обнаруженные публикации (метаданные: тикер, тип, дата публикации, URL, статус обработки, хеш файла).
- `documents` — отдельные документы внутри публикации (PDF, тип отчётности, флаг машиночитаемости).
- `metrics` — извлечённые финансовые показатели (тикер × отчётный период × показатель × значение × источник × тип отчётности).
- `events` — извлечённые существенные факты в структурированном виде.
- `runs` — журнал запусков пайплайна (время старта, время финиша, статус, статистика).

### 10.3. Excel-витрина (вывод)

Файл `e-disclosure.xlsx` с листами:

- `metrics` — плоская таблица: `ticker | reporting_date | period_type | reporting_standard | metric_name | value | currency | unit`.
- `events` — плоская таблица: `ticker | event_date | publication_date | event_type | summary | key_params_json | source_url`.
- `meta` — метаинформация: дата последнего обновления, состав тикеров, версия пайплайна.

### 10.4. Репликация на Google Drive

- После каждого успешного запуска Excel перезаписывается в указанную в конфиге папку Google Drive.
- При перезаписи сохраняется одна и та же ссылка (используется update, а не create) — чтобы ссылка на iPhone не "ломалась".
- Опционально (флаг в конфиге) сохраняются датированные снапшоты в подпапке `archive/` — для аудита.

## 11. Валидация качества (sanity checks)

Применяется к извлечённым LLM показателям до записи в витрину. При срабатывании проверки запись не блокируется, но помечается флагом `qa_warning` с типом нарушения.

### 11.1. Базовые проверки

- **Балансовое уравнение:** `Активы ≈ Капитал + Обязательства` (допуск ±0.5% на округления). Применяется только если все три показателя извлечены.
- **Знаки:** Активы, Выручка — неотрицательные; Чистая прибыль и EBITDA — могут быть отрицательными.
- **Год-к-году:** изменение каждого показателя не должно превышать ±10x относительно предыдущего отчётного периода (флаг "подозрительная динамика", не блокирующий).
- **Согласованность валют:** все показатели в одном отчёте — в одной валюте.
- **Согласованность единиц:** "тыс. руб." vs "млн руб." — при противоречии данные нормализуются к базовой единице или помечаются флагом.

### 11.2. Структурная проверка

- Если для эмитента в отчётном периоде извлечено <50% запрошенных показателей — публикация помечается как "incomplete" и попадает в отдельный отчёт о проблемах извлечения для ручного разбора.

## 12. Запуск

### 12.1. Режимы

- **CLI-команда `update`:** инкрементальный запуск ("кнопка обновить").
- **CLI-команда `run --full-reload`:** полная переобработка с глубиной 3 года.
- **Автоматический запуск:** один раз в сутки через `cron`-задачу (или `systemd` timer), вызывающую `cli update`. Время — задаётся в конфиге, дефолт `04:00` локального времени (после ночной публикации сообщений). Пример crontab-строки идёт в комплекте поставки.

### 12.2. Идемпотентность

Повторный запуск над уже обработанными публикациями не приводит к дублированию записей и не делает повторных вызовов LLM (используется хеш файла + статус в state-БД).

---

## 13. Безопасность секретов

- API-ключи (Anthropic, OpenRouter, опциональный облачный OCR) и OAuth-токены Google Drive хранятся в `.env`, не коммитятся в Git.
- Файл `.env` создаётся из `.env.example` вручную при первичной установке.
- Все секреты подгружаются через Pydantic Settings; код нигде не работает с переменными окружения напрямую.

---

## 14. Логирование и обработка ошибок

- Структурированный JSON-лог в `logs/pipeline.log` с ротацией.
- Уровни: `INFO` для прогресса стадий, `WARNING` для sanity-флагов и неполного извлечения, `ERROR` для падений.
- На падении отдельной публикации — пайплайн **не прерывается**, ошибка фиксируется в `runs`, обработка продолжается со следующей публикации.
- Telegram/email-уведомления **не реализуются** в первой версии (по требованию заказчика).

## 15. Перспектива доступа с iPhone (вне scope первой версии)

Для последующего расширения зафиксированы следующие архитектурные решения, не требующие переработки ядра:

- Бизнес-логика инкапсулирована в модулях, не зависящих от способа вызова.
- State и витрина — в стандартных форматах (SQLite, Excel), читаемых из мобильных приложений.
- Файл на Google Drive автоматически доступен через приложение "Google Диск" или "Numbers" на iPhone — это **минимально достаточный** сценарий мобильного просмотра.
- В будущем поверх ядра можно поднять FastAPI-сервис с REST-эндпоинтами и нативное iOS-приложение. Этот этап оформляется отдельным ТЗ.

## 16. Допущения

- e-disclosure.ru сохраняет текущую структуру URL и форматов раскрытия. При изменениях потребуется обновление модуля `Discoverer` / `Downloader`.
- Объём LLM-вызовов в инкрементальном режиме оценивается как ~50 эмитентов × ~4 публикации в среднем за день максимум ≈ 200 вызовов/сутки в пиковые периоды отчётности; при первоначальном backfill за 3 года — ~3000 вызовов разово.
- Качество извлечения из сканированных PDF существенно зависит от качества скана и может потребовать перехода на облачный OCR — это закладывается как опциональная замена через конфиг.

---

## 17. Принятые решения по ранее открытым вопросам

Зафиксированы по итогам согласования:

1. **Целевая ОС:** Linux. Планировщик — системный `cron` (или `systemd` timer на выбор реализатора), без long-running процесса. Инструкция по установке готовится под Linux.
2. **LLM-доступ:** двухканальный — прямой Anthropic API (приоритет, даёт нативный PDF-input и обычно более высокое качество извлечения) с автоматическим fallback на OpenRouter при недоступности или отсутствии ключа. Оба ключа хранятся в `.env`.

## 18. Что НЕ входит в scope первой версии

- Telegram/email-уведомления.
- Мобильное приложение iOS.
- REST API.
- Динамическое подтягивание состава индекса MOEX.
- Извлечение нефинансовых показателей (ESG, операционные метрики).
- Дашборды и визуализация (только Excel-витрина).
- Алёрты по триггерам (например, "EBITDA упала на 30% YoY").
- Хранение исторических версий извлечений (только последняя успешная по каждой публикации).
