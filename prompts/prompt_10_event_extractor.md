# Промпт 10. Event Extractor (LLM)

## Цель
Структурировать сообщения о существенных фактах: тип события (по справочнику), даты, краткое резюме, ключевые числовые параметры, ссылка на оригинал.

## Контекст из ТЗ
- Раздел 6: «собираются все типы существенных фактов», классификация по справочнику, поля события, конфигурируемый справочник `config/event_types.yaml`.
- Раздел 7.1, п.7.

## Задачи
1. Создать `src/edx/stages/event_extractor/`:
   - `schema.py`:
     - JSON Schema для одного события:
       ```json
       {
         "event_type": "<один из кодов из event_types.yaml>",
         "event_date": "YYYY-MM-DD",
         "publication_date": "YYYY-MM-DD",
         "summary": "1-3 предложения, до 600 символов",
         "key_params": {"<параметр>": <значение | строка>}
       }
       ```
       `event_type` — `enum` с кодами из `EventTypesConfig`.
   - `prompts.py`:
     - System-промпт на русском: задача, краткие описания каждого типа события (брать из `event_types.yaml` → `display_name` + опционально `description`), формат ответа, требование указывать только то, что явно есть в тексте.
   - `service.py`:
     - `EventExtractorService.run(publication)`:
       - применяется к публикациям с `publication_type='event'`.
       - вход:
         - если в публикации есть PDF — взять `text_extract_path` ведущего документа;
         - часто событие — это HTML-страница; в Downloader (промпт 05) уже сохранили её как файл; добавить узкую функцию `html_to_text` (`selectolax` / `BeautifulSoup`) для извлечения текста из HTML-источников. Если такой текст уже есть — использовать его.
       - вызвать LLM-цепочку с JSON-схемой одного события (массив тут не нужен — каждая публикация = одно сообщение о факте).
       - валидировать Pydantic-моделью `EventExtractionResult`.
       - писать в `events_repo.upsert_event(...)`. UNIQUE по `source_publication_id` гарантирует идемпотентность.
       - перевести публикацию в статус `validated` (Validator проверит даты).
2. Поддержать неклассифицируемые события: если LLM не нашёл подходящего кода — использовать резервный код `other` (должен быть гарантирован в `event_types.yaml`; добавить валидацию на уровне Pydantic-схемы конфига).
3. CLI: `edx extract-events --publication-id <id>`.

## Тесты, которые должны проходить
- Юнит-тест построения JSON-схемы из тестового `EventTypesConfig` (snapshot).
- Юнит-тест `html_to_text` на фикстурах HTML-сообщения e-disclosure (`tests/fixtures/events/*.html`) — отсекаются меню/футеры, остаётся основной текст.
- Юнит-тест сервиса с замокированным LLM:
  - валидный ответ → запись в `events`;
  - повторный прогон не плодит дубликатов (UNIQUE по publication);
  - `event_type`, отсутствующий в справочнике, → fallback `other` + warning в логе;
  - длинный `summary` обрезается до 600 символов (или валидация валит, в зависимости от выбранной стратегии — задокументировать).
- Юнит-тест дат:
  - если LLM не вернул `event_date` — берётся `publication.publication_date` с warning.

## Definition of Done
- `events` заполняется только из публикаций с `publication_type='event'`. Метрические отчёты сюда не попадают.
- Изменение справочника типов = правка `event_types.yaml`, кода стадии не касаемся.
- Стадия делит LLM-цепочку с Metric Extractor (никакого дублирования провайдеров).
